# v0.5 — Solitude Runtime Design

> **Philosophy**: Single pillar — Solitude. At any moment, at most one agent exists.
> An agent knows its role, receives its task, works alone, and vanishes.
> No agent has ever met another agent. The orchestrator is not an agent — it is a program.

> **Prerequisite**: v0.1-0.2 (Team presets, contracts, agency-agents provenance).
> v0.3 container lifecycle, v0.4 isolation runtime → **replaced by this design.**

---

## 1. Overview

v0.5 replaces the long-running-container model (N+1 agents, concurrent, network-connected)
with a **sequential pipeline** model: zero containers at rest, one container at a time,
complete solitude between agents.

### Core principles

1. **Solitude over isolation.** Isolation assumes coexistence (agent A _could_ reach agent B, so we block it).
   Solitude assumes non-existence (agent A runs → finishes → vanishes → agent B runs).
   No coexistence → no isolation machinery needed.

2. **One agent, one container, one shot.** Each agent runs in a fresh `docker run --rm`.
   No volumes, no persistent memory, no network ports. The agent gets its task, works,
   writes results, exits. The container, its memory, its /tmp, its process table — all gone.

3. **File-only handoff.** The orchestrator reads the previous agent's output directory and
   writes the next agent's input directory. No network calls between agents, no shared DB,
   no message queue. The handoff contract is a data transformation schema.

4. **Only the orchestrator talks to the user.** The orchestrator is not an AI agent —
   it is a deterministic program that runs the pipeline. It reports progress, collects
   results, and surfaces failures. It never "decides" — only executes the plan.

5. **The pipeline is determined at plan time.** `hermes-fleet plan "goal"` produces
   a complete execution graph (agent sequence, per-agent contracts, data dependencies).
   The runtime (`hermes-fleet run`) executes that graph without deviation.

### Borrowed from Hermes Agent Kanban (v0.12.0)

| Kanban Feature | Fleet Adoption |
|----------------|---------------|
| Task state machine (ready → running → done/blocked) | Pipeline step state machine |
| Dispatch loop: claim → spawn → monitor → reclaim | Scheduler loop: resolve next → run container → collect output |
| Spawn failure circuit breaker (5 failures → auto-block) | Execution retry with threshold |
| Worker log per task | Per-step log under `.fleet/runs/` |
| Heartbeat for long-running tasks | Container-level timeout + periodic output check |
| No-double-claims via SQLite CAS | No-double-execution via lock file |

### What we drop from Kanban

| Kanban Feature | Why Not Needed |
|----------------|---------------|
| SQLite DB for coordination | No concurrent workers. File-based state is sufficient. |
| Task claiming (mutex) | Sequential execution — no contention. |
| Kanban board UI | Pipeline is linear. No board visualization needed. |
| Human-in-loop task creation | Team + pipeline generated by planner automatically. |
| Shared workspace | Each agent gets its own workspace. Previous agent's workspace is read-only reference. |

## 2. Execution Model

### Pipeline

```
plan("iOS 앱 만들어줘")
  → [frontend-developer, backend-developer, qa-tester, devops]
  → file: .fleet/pipelines/current.json

run()
  for each agent in pipeline:
    1. resolve input: last agent's output + goal context → /tmp/fleet/<step>/input/
    2. create workspace: /tmp/fleet/<step>/workspace/
    3. docker run --rm
         -v input:/input:ro
         -v workspace:/workspace
         -e HERMES_AGENT_ID=frontend-developer
         -e HERMES_TASK="Create login page"
         -e HERMES_INPUT_DIR=/input
         -e HERMES_OUTPUT_DIR=/workspace/output
         hermes-agent --task /input/task.md
    4. wait for exit
    5. if exit == 0:
         copy /workspace/output → /tmp/fleet/<step>/output/
         log success, go to next agent
    6. if exit == 1 (failure):
         log error, retry up to N times
    7. if exit == 2 (needs approval):
         save state, notify user, pause
```

### Agent States (per pipeline step)

```
PENDING → RUNNING → DONE
                  → FAILED (retry)
                  → BLOCKED (needs user)
```

No `CREATED`, `ACTIVE`, `IDLE`, `COMPLETED`, `ARCHIVED`. Those were for long-running containers.
A pipeline step is either waiting, running, or finished.

### Data Directory Structure

```
.fleet/
├── fleet.yaml              # unchanged
├── foundation.lock.yaml    # unchanged
├── agency.lock.yaml        # unchanged
├── roles/                  # unchanged
├── presets/                # unchanged
├── pipelines/
│   └── current.json        # current pipeline definition
└── runs/
    └── <run-timestamp>/
        ├── pipeline.json   # snapshot of the pipeline at run time
        └── steps/
            └── <agent-id>/
                ├── input/
                │   ├── task.md
                │   ├── context.json
                │   ├── SOUL.md
                │   └── policy.yaml
                ├── output/
                │   ├── result.json
                │   ├── summary.md
                │   └── files/
                ├── workspace/    (bind-mounted into container)
                ├── exit_code
                └── log.txt
```

## 3. Handoff Contract as I/O Schema

The handoff contract no longer describes a message from agent A to agent B.
It describes **the data transformation the orchestrator must perform**
between agent A completing and agent B starting.

```yaml
# presets/handoffs/frontend-to-backend.yaml
handoff_contract:
  id: frontend-to-backend
  source_agent: frontend-developer
  target_agent: backend-developer

  # What the source agent must produce
  source_output:
    required_fields:
      - files            # list of file paths created/modified
      - api_contract     # API spec for the backend
      - decisions        # design decisions made
    validation:
      - field: api_contract
        min_length: 10

  # How the orchestrator transforms source output into target input
  transform:
    task_description: "Implement API endpoints defined in {source.api_contract}"
    context:
      api_contract: output.files/{source.api_contract}
      frontend_decisions: output.decisions
      previous_output_path: "<step-output-path>"  # set by runtime

  # What the target agent will receive
  target_input:
    required_fields:
      - task_description
      - api_spec
```

The orchestrator reads this contract and:
1. Validates frontend-developer's output against `source_output`
2. Builds backend-developer's input using `transform` template
3. Writes input files to backend-developer's `input/` directory
4. Runs the backend-developer container

## 4. Scheduler (replaces Kanban dispatcher)

The scheduler is a simple Python loop. Not a daemon — it runs once per `hermes-fleet run`.

```python
def run_pipeline(pipeline: Pipeline, run_dir: Path):
    for step in pipeline.steps:
        step.state = "running"
        save_state(run_dir, step)

        # Build input
        input_dir = run_dir / "steps" / step.agent_id / "input"
        output_dir = run_dir / "steps" / step.agent_id / "output"
        workspace_dir = run_dir / "steps" / step.agent_id / "workspace"
        build_input(step, last_output, input_dir)

        # Run container
        exit_code = run_container(
            image=step.image,
            agent_id=step.agent_id,
            task=step.task_description,
            policy=step.policy,
            input_dir=input_dir,
            workspace_dir=workspace_dir,
            timeout=step.timeout_seconds,
        )

        # Collect output
        if exit_code == 0:
            last_output = collect_output(step, output_dir, workspace_dir)
            step.state = "done"
        elif exit_code == 1:
            handle_failure(step, exit_code)
        elif exit_code == 2:
            step.state = "blocked"
            notify_user(...)
            return  # pause pipeline

        save_state(run_dir, step)
```

Key design:

- **No daemon.** Each `run` invocation is a single process that runs the full pipeline or until blocked.
- **State is file-based.** Steps, outputs, logs are all in `.fleet/runs/<id>/`. No SQLite dependency.
- **Blocked state pauses.** If an agent returns exit code 2 (needs user input), the scheduler saves state, prints a message, and exits. User resolves, then `hermes-fleet resume` continues.
- **Retry with backoff.** Configurable per-step, defaults to 3 attempts with exponential backoff.
- **Idempotency.** Each step has a unique run ID. Re-running a step that's already `done` is skipped (lock file in output dir).

### Comparison to Kanban Dispatcher

| Dimension | Kanban Dispatcher | Fleet Scheduler |
|-----------|------------------|-----------------|
| Execution | Daemon (60s interval, forever) | One-shot (run pipeline, exit) |
| State | SQLite DB (WAL mode) | File system (`.fleet/runs/`) |
| Concurrency | Multiple workers via claim_lock | None (sequential) |
| Spawn | `subprocess.Popen(hermes -p ...)` | `subprocess.run(docker run --rm ...)` |
| Failure handling | Circuit breaker (5 failures → auto-block) | Retry with backoff → pause for user |
| Complexity | 2,765 lines (kanban_db.py) | ~400 lines expected |

## 5. Specialized Agent via agency-agents

This is Fleet's differentiator. Each container run includes:

1. **SOUL.md** — Role identity from agency-agents upstream with provenance metadata.
   The agent knows exactly who it is, what it does, and what it doesn't do.

2. **policy.yaml** — Task types, filesystem permissions, command allow/deny lists.
   Generated from the role contract + permission preset.

3. **Handoff contract (I/O schema)** — The agent knows what it must produce.
   The contract is injected as `/input/handoff-contract.yaml` so the agent
   can validate its own output before exiting.

### Example: frontend-developer container

```
docker run --rm \
  -v .fleet/runs/<id>/steps/frontend-developer/input:/input:ro \
  -v .fleet/runs/<id>/steps/frontend-developer/workspace:/workspace \
  hermes-agent:latest \
  --profile frontend-developer \
  --task /input/task.md
```

Inside the container, the agent sees:
- `/input/task.md` — what to do: "Create login page"
- `/input/SOUL.md` — who I am: "Frontend Developer. I write UI code."
- `/input/policy.yaml` — my boundaries: "allowed_tasks: [ui, styling], forbidden_tasks: [backend, database]"
- `/input/context.json` — what the previous agent did, what files exist
- `/input/handoff-contract.yaml` — what I must produce to pass to the next agent

The agent works, writes results to `/workspace/output/`, and exits.
Exit code 0 = done, 1 = failed, 2 = blocked (needs user).

## 6. Team Composition (Planner)

The planner evolves from keyword matching to a **rule-based team composer**:

```yaml
# team rules derived from Contract Net Protocol + role analysis
team_rules:
  - pattern: ["ios", "iphone", "swift"]
    team: iphone-app
    pipeline: [ux-designer, frontend-developer, backend-developer, qa-tester, devops]

  - pattern: ["saas", "web app", "dashboard"]
    team: saas-medium
    pipeline: [frontend-developer, backend-developer, database-architect, qa-tester, devops]
```

Each team preset now includes a `pipeline` field — the execution order.
The planner additionally applies dependency analysis:
- If the user says "existing repo" → frontend-developer runs first (reads codebase)
- If the user says "from scratch" → ux-designer runs first (generates spec)

## 7. CLI Surface

### New commands

| Command | Description |
|---------|-------------|
| `hermes-fleet plan "<goal>"` | Generate pipeline plan. Shows agent sequence with contracts. |
| `hermes-fleet run` | Execute current pipeline. Runs containers sequentially. |
| `hermes-fleet run --step backend-developer` | Resume from a specific step. |
| `hermes-fleet status` | Show current pipeline progress (which steps done/pending/blocked). |
| `hermes-fleet logs <step>` | Show logs from a completed/failed step. |

### Removed commands

| Command | Reason |
|---------|--------|
| `hermes-fleet up` | No containers to start. |
| `hermes-fleet down` | No containers to stop. |
| `hermes-fleet restart` | No containers to restart. |
| `hermes-fleet volume wipe/snapshot/restore` | No persistent volumes. |
| `hermes-fleet agent idle/wake/archive` | No long-running agents. |
| `hermes-fleet handoff validate` | Validation happens automatically in scheduler. |
| `hermes-fleet test safe-defaults` | Merged into `validate` (already done). |
| `hermes-fleet generate` | Replaced by `plan` + `run`. `plan` generates the pipeline. |

### Kept with modifications

| Command | Modification |
|---------|-------------|
| `hermes-fleet init` | Pipeline dirs instead of generated dirs. |
| `hermes-fleet plan` | Now produces full pipeline with data contracts. |
| `hermes-fleet validate` | No longer calls safe-defaults (those checks are obsolete). |

## 9. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Container startup latency (2-10s per step) | Pipeline feels slow | Pre-pull image at `plan` time. Cache in CI. |
| Agent exit code discipline | Scheduler can't distinguish failure from blocked | Enforce exit code convention via entrypoint wrapper. |
| Large output files (100MB+) | Workspace disk usage | Clean up old runs (`hermes-fleet clean`). Configurable retention. |
| Hermes Agent batch mode doesn't exist | Cannot run non-interactively | Fleet provides entrypoint script that feeds task + reads output. |
| User wants real-time streaming | No live output | Not a goal. Pipeline is fire-and-forget. Status is post-hoc. |
