# v0.4 — Isolation Runtime Design

> **Philosophy**: Single pillar — Isolation. Every agent has its own room.
> Only the orchestrator holds all the keys.
>
> **Prerequisite**: v0.3 (Container Lifecycle) — fleet up/down/status/logs/restart.
> v0.4 makes the runtime enforce what v0.1-v0.3 only declared.

---

## 1. Overview

v0.4 adds three layers of isolation enforcement at runtime:

| Layer | What | How |
|-------|------|-----|
| Memory | Agent A cannot read agent B's memory | Per-agent Docker volumes, lifecycle management |
| Communication | No direct agent-to-agent or agent-to-user | Gateway routing via orchestrator only |
| Network | Agents default to no network, exceptions via approval | Role-based policy, orchestrator-proxied temp access |

Plus supporting infrastructure:
- Agent lifecycle state machine (CREATED → ACTIVE → IDLE → COMPLETED → ARCHIVED)
- Handoff contract runtime validation
- Token budget per session

## 2. Memory Isolation

### Current State (v0.3)
Each agent gets a named Docker volume (`hermes-fleet-{agent_id}-{team}_data`) mounted at `/opt/data`. Volumes persist across restarts but are never explicitly cleaned up.

### v0.4 Changes

**Volume lifecycle management** — add CLI commands and volume lifecycle to runner.py:

- `hermes-fleet volume wipe <agent>` — delete agent's volume (all memory lost)
- `hermes-fleet volume snapshot <agent>` — create a snapshot of agent memory
- `hermes-fleet volume restore <agent> <snapshot>` — restore from snapshot

**Volume naming convention** (unchanged):
```yaml
volumes:
  agent_name_data:
    driver: local
```

**Design decisions**:
- No cross-agent volume mounts. Each agent sees only its own volume.
- Snapshot is a `docker run --rm -v ... tar czf` execution, stored in `.fleet/snapshots/`
- `volume wipe` is `docker volume rm`. Destructive, requires confirmation.

### Files to modify:
- `src/hermes_fleet/runner.py` — add `volume_wipe()`, `volume_snapshot()`, `volume_restore()`
- `src/hermes_fleet/cli.py` — add `hermes-fleet volume` subcommand

## 3. Communication Isolation

### Current State (v0.3)
Agents generate Docker Compose files. All agents share networks. No gateway-level routing exists. A user could theoretically talk to any agent's Hermes gateway directly.

### v0.4 Changes

**Gateway routing rule**: The orchestrator is the only service exposed to user traffic. Sub-agents' gateway ports are not published.

In generated Docker Compose:
```yaml
services:
  orchestrator:
    ports:
      - "127.0.0.1:PORT:8080"  # only orchestrator port exposed
  backend-developer:
    ports: []  # no ports published
    networks: [fleet-no-net]  # no external network
```

**Implementation**:
- `_select_network()` in `docker_compose.py` already distinguishes orchestrator from sub-agents
- Add port publishing logic: orchestrator gets ports, sub-agents get empty ports
- Add `HERMES_GATEWAY_ORCHESTRATOR_ONLY=true` env var for gateway-level enforcement

**Design decisions**:
- Port isolation is Docker-level, not application-level
- The future gateway plugin (v0.5) will add message-level routing enforcement
- v0.4 ensures sub-agents are unreachable at network level

### Files to modify:
- `src/hermes_fleet/docker_compose.py` — add port publishing logic per role type

## 4. Network Isolation

### Current State (v0.3)
Policy model supports `network_access` field with values: `none`, `control_plane_only`, `package_registry`, `web_readonly`. Three Docker networks exist: `fleet-no-net` (internal), `fleet-control-plane` (internal), `fleet-web` (external).

### v0.4 Changes

**Four network modes**:

| Mode | Docker Network | Internet | Fleet Agents | Use Case |
|------|---------------|----------|--------------|----------|
| `isolated` | fleet-no-net (internal) | No | No | Security reviewer, QA |
| `control-plane` | fleet-control-plane (internal) | No | Yes (orchestrator only) | Code reviewer |
| `proxy` | fleet-control-plane (internal) | Via orchestrator | Yes | Frontend/backend dev |
| `extern` | fleet-web (bridge) | Yes | Yes | DevOps, architect |

**Default by role** (set at team composition time, stored in fleet.yaml):
```yaml
network_policy:
  security-reviewer: isolated
  qa-tester: isolated
  frontend-developer: proxy
  backend-developer: proxy
  devops: extern
  orchestrator: control-plane
```

**Temporary network access flow** (infrastructure for v0.5 orchestrator):

```
Agent A: "I need npm access (registry.npmjs.org:443)"
  → Event bus (agent → orchestrator)
  → Orchestrator: reviews request, checks policy
  → User: "Agent A requests temp npm access. Approve?"
  → User approves (duration: 15m)
  → Orchestrator: applies temp iptables rule or proxy config
  → Timer starts. On expiry: rule auto-revoked.
```

In v0.4, the event bus and proxy infrastructure is built. The orchestrator integration (v0.5) will use it.

### Files to modify/ create:
- Create `src/hermes_fleet/network.py` — network policy resolution, temp access management
- `src/hermes_fleet/docker_compose.py` — use network policy from fleet.yaml, not just role preset
- `src/hermes_fleet/contracts.py` — add `NetworkConfig` contract with time-limited access model

## 5. Agent Lifecycle State Machine

### Current State (v0.3)
Container-level: running/stopped/crashed. No agent-level state.

### v0.4 State Machine

```
CREATED → ACTIVE → IDLE → COMPLETED → ARCHIVED
                  ↑        ↓
                  └── ACTIVE (wake)
```

| State | Meaning | Docker Container |
|-------|---------|-----------------|
| CREATED | Config generated, not yet started | Not running |
| ACTIVE | Agent is working on tasks | Running |
| IDLE | Agent not accepting tasks, memory preserved | Running (idle) |
| COMPLETED | Agent finished its role in the project | Stopped |
| ARCHIVED | Agent removed from fleet, volume may be wiped | Removed |

**Implementation**:
- Add `agent_state` field to `FleetConfigContract`
- Add CLI commands: `hermes-fleet agent idle <agent>`, `hermes-fleet agent wake <agent>`, `hermes-fleet agent archive <agent>`
- IDLE agents reject messages at gateway level (env var `HERMES_AGENT_STATE=idle`)

### Files to modify:
- `src/hermes_fleet/contracts.py` — extend `FleetConfigContract` with agent state
- `src/hermes_fleet/cli.py` — add `hermes-fleet agent` subcommand group
- `src/hermes_fleet/runner.py` — connect CLI to Docker container state

## 6. Handoff Contract Runtime Validation

### Current State (v0.3)
Handoff contracts are validated at generation time (cross-references). No runtime validation exists.

### v0.4 Changes

Add `hermes-fleet handoff validate` command that validates a handoff document against its contract at runtime:

```bash
hermes-fleet handoff validate <from-agent> <to-agent> <handoff-file>
```

Validation checks:
- `from_roles`: sender agent must be in contract's `from_roles`
- `allowed_next_roles`: receiver agent must be in `allowed_next_roles`
- `required_fields`: all required fields must be present in the handoff document
- `validation_rules`: field-specific rules (min_length, enum, regex, etc.)

**Implementation**:
- Runtime validation uses the same `HandoffContract` Pydantic model
- Add `validate_handoff_doc()` function that checks a handoff doc dict against the contract
- CLI returns structured results (passed/failed per rule)

### Files to modify:
- `src/hermes_fleet/contracts.py` — add `validate_handoff_doc()` function
- `src/hermes_fleet/cli.py` — add `hermes-fleet handoff validate` command

## 7. Token Budget

### Current State (v0.3)
No token budget enforcement.

### v0.4 Changes

Add `max_iterations_per_session` to fleet.yaml:
```yaml
token_budget:
  default: 50
  per_agent:
    orchestrator: 200
    frontend-developer: 100
    security-reviewer: 30
```

Implementation:
- Field in `FleetConfigContract`
- Passed as env var `HERMES_MAX_ITERATIONS=<n>` to each container
- CLI: `hermes-fleet config set token_budget.per_agent.frontend-developer 150`

### Files to modify:
- `src/hermes_fleet/contracts.py` — add token_budget to FleetConfigContract
- `src/hermes_fleet/docker_compose.py` — pass budget as env var to services
- `src/hermes_fleet/cli.py` — extend config set for nested keys

## 10. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Docker volume wipe is destructive | Data loss | Require `--force` flag, add `volume snapshot` before wipe |
| Port publishing breaks existing deployments | Backward incompatibility | New fleet.yaml field `expose_ports: false` (default: false for sub-agents) |
| Agent lifecycle complicates status reporting | Confusion | `hermes-fleet status` shows both container state and agent state |
| Network isolation blocks legitimate traffic | User frustration | Temp access flow, clear error messages explaining how to request access |
| Token budget too low | Agents can't complete tasks | Budget per agent per session, warning at 80% usage |
