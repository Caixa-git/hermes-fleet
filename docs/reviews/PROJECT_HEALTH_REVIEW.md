## Detailed Findings

### 🔴 Finding 1: Optional agent role files do not exist

**File**: `presets/teams/saas-medium.yaml` (lines 18-21)

The YAML declares three `optional_agents`:
```yaml
optional_agents:
  deployer: false
  growth-marketer: false
  customer-support-specialist: false
```

But none of these have corresponding role files:
- `presets/roles/deployer.yaml` → ❌ **missing**
- `presets/roles/growth-marketer.yaml` → ❌ **missing**
- `presets/roles/customer-support-specialist.yaml` → ❌ **missing**

**Impact**: Low — these are disabled by default (`false`), so no code path hits them. However, if a future flag like `hermes-fleet generate --enable deployer` is added, it would break because `compose_policy("deployer")` would return an empty policy.

**Recommendation**: Either:
- (a) Create minimal role files for all three optional agents, even if they're never generated by default
- (b) Remove `optional_agents` from the YAML until these roles are ready to be designed
- (c) Add a validation check (test) that if `optional_agents` lists a name, a corresponding role file must exist

### 🟡 Finding 2: Generator ignores `optional_agents` field

**File**: `src/hermes_fleet/generator.py` (line 40)

```python
agents = team_def.get("agents", [])
```

The generator only reads the `agents` list. It completely ignores the `optional_agents` dict. This means:
- The `--team` override from CLI cannot enable optional agents
- There is no `--enable` or `--include` flag
- The optional_agents field is **declarative metadata only** — it has no code path

**Impact**: Low. The field is properly documented in SPEC.md section 4.2 and the YAML is clear. But this is a gap between declaration and implementation.

**Recommendation**: Either:
- (a) Add a comment in generator.py explaining that optional_agents is reserved for v0.2+
- (b) In v0.2, add a `--include <agent>` flag that merges optional agents with active=true into the agents list

### 🟡 Finding 3: SPEC.md section 6 missing `schema_worktree_rw` preset

**File**: `SPEC.md` (section 6, permission presets table)

The SPEC table lists 7 presets:
```
orchestrator_safe, repo_readonly, docs_rw_repo_ro,
frontend_worktree_rw, backend_worktree_rw,
readonly_no_network, test_runner
```

But there are actually **8** preset files:
```
backend_worktree_rw, docs_rw_repo_ro, frontend_worktree_rw,
orchestrator_safe, readonly_no_network, repo_readonly,
schema_worktree_rw, test_runner
```

`schema_worktree_rw` exists in the filesystem but is **missing from the SPEC table**. It is referenced by `database-architect` role and is fully functional — just undocumented in the SPEC.

**Impact**: Low. The preset works correctly. But the SPEC is the authoritative reference document, and a user reading it would not know `schema_worktree_rw` exists.

**Recommendation**: Add one row to the SPEC table:
```
schema_worktree_rw | own_worktree_rw | true | DATABASE_URL_DEV | package_registry
```

### 🟡 Finding 4: README says 21 checks, ROADMAP says 22

**File**: `README.md` (What v0.1 Does table) vs `ROADMAP.md` (v0.1 features)

| Document | Claim |
|----------|-------|
| README | "Validate generated config against 21 safe-default rules" |
| ROADMAP | "Safe-defaults validator (22 checks across 16 check functions)" |
| Actual | 16 check functions, totaling 22 individual checks (7 Docker + 15 local) |

The discrepancy likely comes from a historical count change. The README example also shows `Passed: 20  Failed: 0  Skipped: 1` which would be 21 total — but the actual code generates 22 sub-checks (the missing 1 is the `docker_security_checks` wrapper itself, which doesn't appear as a result).

**Impact**: Cosmetic. No functional impact.

**Recommendation**: Sync README to match ROADMAP (22 checks). Update the README example to reflect 22 total.
